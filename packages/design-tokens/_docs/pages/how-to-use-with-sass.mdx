# Access tokens via SCSS

## 1. Set up a KaizenProvider

Ensure you have followed the steps in <LinkTo pageId="components-kaizen-provider-installation">KaizenProvider</LinkTo> to set up your application for using themes.

## 2. Start using design tokens

There are 2 different ways you can utilise the tokens:

1. Via CSS vars directly (Recommended)
2. Importing SCSS vars from `@kaizen/design-tokens`

### Using CSS Vars

```scss
.myCustomClassName {
  background-color: var(--bg-secondary);
}
```

### Importing SCSS vars

- Import the CSS variable versions of the tokens
- Use these tokens in your CSS. They will be compiled to CSS variables (e.g. `$bg-primary` will become `var(--bg-primary)`)

Tokens are split across category-scoped entrypoints — import only the ones you need. There is no single "everything" SCSS import; each category is a separate file:

## 3. Be aware of the limitations of CSS variables in SASS

### Compile-time functions like `mix()` or `darken()` won't work

**It won't be possible to do things like:**

<CodeBlock
  language="scss"
  code={`.myClass {
  color: mix(white, $text-primary, 90%);
}`}
/>

Because the exposed SASS variables contain values such as `var(--text-primary)`, the SASS compiler won't be able to calculate anything that relies on concrete values such as HEX strings.

Usually when we've used functions like `mix` or `shade` it has been to lighten or darken a token by a certain amount. We recommend you find the nearest token that already has the tone you need — pick the semantic token for that purpose (e.g. `$text-secondary`) rather than computing one.

## 4. Avoid adding opacity to colour tokens

> **⚠️ Don't manipulate the opacity of a colour token.** Reach for a solid token
> that already has the value you want instead.

Applying alpha to a colour (e.g. `rgba($color-purple-800-rgb, 0.5)`) is discouraged:

- **Accessibility.** A translucent colour composites with whatever sits behind it, so its effective contrast depends on the background and is no longer predictable. This breaks contrast guarantees and is a common cause of WCAG failures.
- **Semantic tokens are solid.** Our semantic tokens (`text-secondary`, `text-tertiary`, etc.) deliberately map to **solid** primitives — they replace the old `purple-800` alpha hacks. Using `rgba()` on a token side-steps that system and reintroduces the problem we removed.

**Instead, pick the nearest solid token.** If you previously dimmed a colour with opacity, choose the semantic token that lands at the tone you need — e.g. use `$text-tertiary` rather than applying alpha to `$text-primary`.

<details>
<summary>If you genuinely still need alpha (rare)</summary>

Only **primitive** colours expose a `-rgb` sibling variable — semantic tokens do not — so this escape hatch necessarily drops down to a primitive. Plain `rgba($color-gray-200, 0.5)` won't work either: neither SASS nor the browser can resolve `rgba(var(--color-gray-200), 0.5)`. The `-rgb` sibling points at a comma-separated channel list (`246, 246, 246`) for use inside `rgba()`:

<CodeBlock language="scss" code={`@import '~@kaizen/design-tokens/sass/color.scss';

.myClass {
background-color: rgba($color-gray-200-rgb, 0.5);
}`} />

`rgba($color-gray-200-rgb, 0.5)` compiles to `rgba(var(--color-gray-200-rgb), 0.5)`, which the browser resolves to `rgba(246, 246, 246, 0.5)`. Treat this as a last resort, not the default.

</details>
